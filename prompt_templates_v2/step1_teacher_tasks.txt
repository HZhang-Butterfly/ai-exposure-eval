# Task Design Principles (NEW — must follow)

1. **Complete Context**: Every task MUST include concrete data in reference_context:
   - For financial/data tasks: provide actual numbers, tables, or transaction records (at least 5–10 rows)
   - For analytical tasks: provide real datasets, not just descriptions
   - For writing tasks: provide source materials to summarize/analyze
   The student should NEVER need to invent data — all inputs must be given.

2. **Edge Cases & Boundary Conditions**: At least {edge_case_count} of the {num_tasks} tasks must include a tricky edge case or exception scenario. Examples:
   - An unusual entry that breaks a normal pattern (e.g., negative value, missing field, duplicate)
   - A policy conflict or ambiguity the student must resolve
   - A multi-step dependency where step 2 changes if step 1 reveals an anomaly
   Tag these tasks with "has_edge_case": true.

3. **Difficulty Levels**: Assign each task a difficulty level:
   - "easy": straightforward application of standard knowledge (expect 4–5 from a competent model)
   - "medium": requires synthesis of multiple concepts or multi-step reasoning (expect 3–4)
   - "hard": involves ambiguity, trade-offs, or uncommon edge cases (expect 2–3)
   Distribution: at least 1 easy, at least 1 hard, rest medium.

4. **Multi-Step Tasks**: At least half the tasks should require 2+ distinct reasoning or work steps (e.g., "first calculate X, then use X to derive Y, finally recommend Z").
